# Agent 入门：只理解两个主题

这份文档只讲两个问题：

1. MOMO Scholar 当前实现为什么不是 Agent？
2. Agent 控制循环是什么？

暂时不学习 Planning、Memory、Multi-Agent 等其他概念。

## 主题一：当前实现为什么不是 Agent？

### 当前程序怎么工作

当前核心流程可以简化成：

```text
用户输入研究问题
→ 搜索 arXiv 论文
→ 删除重复论文
→ 保存论文数据
→ 生成初始报告
```

程序每次都按照这个固定顺序运行。它不会自己判断：

- 搜到的论文是不是太少；
- 要不要换一个搜索词；
- 要不要再搜索一次；
- 当前资料是否已经足够；
- 下一步应该使用哪个工具。

这些步骤由程序员提前写好了，所以当前实现叫 **Workflow（工作流）** 更准确。

### 用生活中的例子理解

Workflow 像一张固定的办事清单：

```text
打开网站 → 搜索论文 → 保存结果 → 生成报告
```

无论中间发生什么，它都按照清单继续执行。

Agent 更像能够根据情况作决定的研究助理：

```text
先搜索论文
→ 发现只有一篇相关论文
→ 判断资料不足
→ 换关键词继续搜索
→ 判断证据已经足够
→ 停止搜索并生成报告
```

最重要的区别不是“有没有调用大模型”，而是：

> 系统能不能根据当前情况，动态决定下一步做什么。

### 使用 LLM 就是 Agent 吗？

不是。假设程序固定执行：

```text
搜索论文 → 调用 LLM 总结 → 保存报告
```

即使中间调用了 LLM，整个执行顺序仍然是提前写死的，所以它仍然可以只是 Workflow。LLM 是一个能力组件，不等于 Agent。

### 当前项目为什么先实现 Workflow？

很多基础工作不需要 Agent 决定：

- XML 按确定规则解析；
- 论文字段按 Schema 校验；
- 文件按确定格式保存；
- 重复论文按明确规则处理。

这些步骤使用普通代码更加稳定，也更容易测试。后续可以把真正需要判断的部分交给 Agent，例如：

- 搜索结果是否足够；
- 是否需要修改查询词；
- 下一步应该搜索、读取 PDF，还是生成报告；
- 现有证据是否足以支持结论。

### 面试回答示例

面试官问：

> 你的 MOMO Scholar 现在是 Agent 吗？

可以回答：

> 当前实现首先是一个确定性的论文处理 Workflow。它按照固定顺序完成 arXiv 检索、数据标准化、去重和报告输出，还没有根据中间结果自主选择下一步行动。我先用 Workflow 验证数据模型和完整流程，因为 XML 解析、Schema 校验、文件写入等步骤使用普通代码更加稳定。后续会把查询调整、工具选择和证据充分性判断等需要语义决策的部分交给 Agent。

### 可能追问

**为什么不一开始全部做成 Agent？**

Agent 的输出不完全稳定，调试和测试成本更高。确定性步骤没有必要交给模型，只在需要语义判断时使用 Agent。

**Agent 的自主程度是不是越高越好？**

不是。自主程度越高，成本、延迟和不可控风险也越高。系统应限制工具、预算、循环次数和可执行动作。

## 主题二：Agent 控制循环

### 什么是控制循环？

Agent 通常会不断重复下面的过程：

```text
查看当前状态
→ 决定下一步
→ 执行动作
→ 查看执行结果
→ 更新状态
→ 决定继续还是停止
```

常见英文表示是：

```text
State → Decide → Act → Observe → Update → Stop or Continue
```

不用死记英文，先理解它是在“做一步以后，根据结果再决定下一步”。

### 放进 MOMO Scholar 中理解

假设用户提出问题：

```text
LLM Agent 如何减少科研综述中的幻觉？
```

第一次循环：

```text
当前状态：还没有论文
决定：先搜索相关论文
行动：调用 arXiv 搜索工具
观察：只得到一篇相关论文
更新：保存论文，并记录资料不足
继续：需要再次搜索
```

第二次循环：

```text
当前状态：只有一篇论文
决定：修改关键词
行动：使用新关键词搜索
观察：得到五篇新论文
更新：保存并去重
继续：需要检查论文内容
```

第三次循环：

```text
当前状态：已有六篇论文
决定：资料已经够用
行动：生成初始报告
观察：报告成功生成
更新：记录报告路径
停止：任务完成
```

这就是控制循环。

### 控制循环中的六个词

- `State`：Agent 当前知道什么、已经做过什么，例如问题、论文、证据、错误和剩余预算。
- `Decide`：根据状态判断下一步做什么。
- `Act`：调用受到系统允许的工具。
- `Observe`：获得工具结果、空结果或错误。
- `Update`：把新结果和错误保存回状态。
- `Stop`：成功、预算耗尽或无法继续时结束任务。

### 为什么一定要有停止条件？

没有停止条件，Agent 可能不断重复：

```text
搜索 → 不满意 → 再搜索 → 仍不满意 → 再搜索……
```

因此程序必须强制限制它，例如：

- 最多执行 10 次；
- 最多运行 5 分钟；
- 最多消耗指定 Token；
- 相同工具和参数不能连续重复；
- 证据达到要求后必须停止。

这些限制应该由普通代码控制，不能只依赖 LLM 自觉遵守。

### 面试回答示例

面试官问：

> 如果把当前 Pipeline 改成 Agent，你会怎么做？

可以回答：

> 我会先增加显式运行状态，记录用户问题、已有论文、证据、历史动作、错误和剩余预算。然后加入决策步骤，根据当前状态选择下一项受控工具。工具执行结果作为 Observation 写回状态，再进入下一轮决策。程序层面设置最大循环次数、预算和重复动作检测，确保 Agent 能够停止，也不会无限调用工具。

### 可能追问

**如何防止 Agent 死循环？**

设置最大循环次数、时间和 Token 预算；保存历史动作；检测重复工具参数；明确成功和失败停止条件。

**工具调用失败怎么办？**

把错误也作为 Observation 写回状态。超时等临时错误可以有限重试；参数错误等永久错误应修改行动或停止。

**为什么状态不能只放在 Prompt 中？**

Prompt 中的状态难以校验、查询和恢复。重要状态应该结构化保存，Prompt 只读取当前决策所需的部分。

## 最后只记住三句话

1. 当前 MOMO Scholar 是固定顺序执行的 Workflow，还不是完整 Agent。
2. Agent 的关键是根据状态和工具结果动态决定下一步，而不是简单调用 LLM。
3. Agent 通过“状态、决策、行动、观察、更新、停止”组成控制循环。
