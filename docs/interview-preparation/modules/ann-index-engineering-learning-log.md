# ANN、HNSW、IVF 与检索评测学习记录

## 学习目标

理解精确向量搜索与近似最近邻搜索的区别，掌握 HNSW、IVF、Recall@K，以及 ANN 技术评测和 RAG 业务评测在面试中所需的最低知识。

本记录不展开图搜索公式、聚类推导和具体参数调优。

## 1. 从 MOMO Scholar 的精确搜索开始

`InMemoryVectorStore.search()` 会遍历所有记录：

```python
for record in self._records.values():
    cosine = _cosine_similarity(query, record.embedding)
```

若有 `n` 个 Chunk，每个向量是 `d` 维，相似度计算约为 `O(n × d)`。这种与所有向量逐一比较的方式称为 Exact Search，也叫暴力搜索或全量扫描。

优点是实现简单，不会因为近似索引漏掉数学意义上的最近邻；缺点是数据量增长后查询延迟随之增长。

## 2. ANN 是什么

ANN 全称 Approximate Nearest Neighbor，即近似最近邻搜索。ANN 不是单个算法，而是一类避免扫描全部向量的方法。

```text
精确搜索：扫描所有向量，得到精确 Top-K
ANN：搜索部分候选，快速得到近似 Top-K
```

其核心取舍是：牺牲少量最近邻召回准确率，换取更低的查询延迟。HNSW 和 IVF 都是常见的 ANN 方法。

## 3. HNSW

HNSW 全称 Hierarchical Navigable Small World，即分层可导航小世界图。

最低限度的理解是：

> HNSW 把向量组织成分层的邻居关系图，从高层快速定位大致区域，再逐层进入局部范围寻找最近邻。

可以类比地图导航：

```text
全国范围 → 城市 → 街区 → 附近地点
```

HNSW 通常能降低查询延迟，但会带来：

- 额外的图索引内存；
- 索引构建时间；
- 写入与索引维护成本；
- 速度和召回率之间的参数调优；
- 近似结果可能无法完全复现精确搜索。

需要特别纠正：HNSW 通常不是让查询变慢，而是可能让索引构建和写入维护变慢。

## 4. IVF

IVF 全称 Inverted File Index。在向量检索中，可以直观理解为：

> 先通过聚类把向量空间分成多个区域，查询时只搜索最可能相关的若干区域。

```text
所有向量 → 聚类分区
Query → 找到接近的分区 → 在部分分区内搜索 → Top-K
```

搜索区域少时速度更快，但可能漏掉相邻区域中的结果；搜索区域多时召回率可能提高，但查询成本也会上升。

## 5. 三种方式的比较

| 方式 | 基本思路 | 主要优点 | 主要代价 |
|---|---|---|---|
| 精确搜索 | 与所有向量比较 | 结果精确、实现简单 | 数据大时慢 |
| HNSW | 沿分层邻居图导航 | 查询快、召回通常较好 | 内存、构建和维护成本 |
| IVF | 先分区，再搜索部分区域 | 减少搜索范围 | 分区范围影响召回率 |

最短记忆：

```text
ANN：近似搜索这一类方法
HNSW：建立分层邻居地图
IVF：先给向量分区，再搜索相关区域
```

## 6. 小规模项目是否需要立即使用 HNSW

若 MOMO Scholar 只有约 5000 个 Chunk，且精确检索延迟满足要求，就不应立即引入 HNSW：

- 全量扫描仍能满足延迟目标；
- 精确搜索结构简单、结果确定；
- HNSW 增加内存占用；
- HNSW 增加建索引、写入维护和调参成本；
- 当前规模下，复杂索引的收益可能小于成本。

是否引入 ANN 应根据向量数量、并发量、P50/P95/P99 延迟、QPS、召回质量和资源成本决定，而不是因为向量数据库支持 HNSW 就提前使用。

面试表达：

> 当前只有约 5000 个 Chunk，而且精确检索延迟满足要求，所以我不会立即引入 HNSW。HNSW 能加速大规模查询，但会增加内存、索引构建、写入维护和参数调优成本。现阶段使用精确检索更简单，也能保证结果准确；等数据量、并发量或 P95 延迟达到瓶颈后，再根据基准测试决定是否引入。

## 7. Recall@K 与 Precision@K

假设人工标注的相关 Chunk 为：

```text
[A, B, C, D]
```

系统 Top-5 为：

```text
[A, X, B, Y, C]
```

系统找回了 `A`、`B`、`C` 三个相关 Chunk。

```text
Recall@5
= Top-5 找回的相关结果数 / 全部相关结果数
= 3 / 4
= 75%

Precision@5
= Top-5 中的相关结果数 / Top-5 结果总数
= 3 / 5
= 60%
```

记忆：

```text
Recall：该找的，有没有漏掉？
Precision：找回的，有多少是真的相关？
```

## 8. ANN 技术评测

ANN 技术评测回答：

> HNSW 或 IVF 的近似结果，与全量精确搜索的结果有多接近？

其中：

```text
HNSW：被评价对象
精确搜索：技术参照基准
```

例如：

```text
精确搜索 Top-5：[A, B, C, D, E]
HNSW Top-5：    [A, B, C, D, X]
```

HNSW 找回精确搜索结果中的四个：

```text
ANN Recall@5 = 4 / 5 = 80%
```

它只说明 HNSW 找回了多少数学意义上的精确最近邻，不说明这些 Chunk 一定能够回答用户问题。

## 9. RAG 业务评测

RAG 业务评测回答：

> 检索结果是否找到了真正能够回答问题或支持结论的证据？

参照标准是人工标注或高质量评测集：

```text
搜索结果 vs 人工标注的相关 Evidence
```

HNSW 和精确搜索都可以成为业务评测对象：

```text
HNSW vs 人工标注 → HNSW 结果的业务质量
精确搜索 vs 人工标注 → 精确搜索结果的业务质量
```

因此：

> 精确向量搜索只是 ANN 的技术参照，不是 RAG 业务上的标准答案。

即使 HNSW 完全复现精确搜索，如果 Embedding 或 Chunking 不适合当前问题，业务检索质量仍可能很差。

最短记忆：

```text
HNSW vs 精确搜索：测“近似得准不准”
搜索结果 vs 人工标注：测“业务上对不对”
```

## 10. 两层评测如何定位问题

```text
HNSW 与精确搜索差异很大
→ 可能是 ANN 索引或参数问题

HNSW 与精确搜索一致，
但都找不到人工标注证据
→ 可能是 Chunking、Embedding、Query 表达、
   混合检索或 Rerank 的问题
```

生产评估不能只看查询是否变快，还应关注：

- P50、P95、P99 查询延迟与 QPS；
- ANN 相对精确搜索的 Recall@K；
- RAG 相对人工标注的 Recall@K；
- 内存占用与索引大小；
- 索引构建时间；
- 新数据写入吞吐；
- 索引加载和恢复时间。

## 11. 与 MOMO Scholar 的关系

当前 `InMemoryVectorStore` 是测试与小规模验证使用的精确搜索参考实现。未来可以增加 `PgVectorStore`、`QdrantVectorStore` 等适配器，并在适配器内部根据数据规模选择精确搜索、HNSW 或 IVF。

上层 `VectorRetriever` 不需要知道底层使用哪种索引算法，这是 `VectorStore` 接口抽象的重要价值。

## 12. 本轮结论

1. ANN 是近似最近邻搜索这一类方法。
2. HNSW 使用分层邻居图，IVF 使用向量分区。
3. `InMemoryVectorStore` 当前执行全量精确扫描。
4. 小数据且延迟可接受时，不必提前引入 ANN。
5. HNSW 通常降低查询延迟，但增加内存、建索引、写入维护和调参成本。
6. Recall@K 关注相关结果是否被漏掉，Precision@K 关注返回结果中有多少真正相关。
7. ANN 技术评测用精确搜索评价近似算法。
8. RAG 业务评测用人工标注证据评价搜索结果。
9. 精确搜索保证数学上的最近邻，但不能保证结果一定支持最终答案。
