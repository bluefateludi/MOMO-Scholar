## 模块目标

用户输入研究问题后，系统从 arXiv 获取候选论文，将第三方数据转换为统一的 `Paper` 模型，去除重复结果，并在下一步通过 Pipeline 生成结构化数据和初始 Markdown 报告。

## 数据流

```text
研究问题
→ arXiv 查询参数
→ HTTP GET
→ Atom XML
→ XML 解析
→ Paper 模型
→ 标题标准化与去重
→ papers.json
→ report.md
```

## 原理一：HTTP 与 arXiv API

当前 `search_arxiv()` 使用 HTTP GET 调用 arXiv API，通过 `urlencode()` 构造查询参数，并设置请求超时。`raise_for_status()` 负责把非成功状态码转换为明确的 HTTP 异常，防止后续把服务端错误误判为 XML 解析问题。

需要深入掌握：

- GET 请求、查询参数和 URL 编码。
- HTTP 状态码与网络异常的区别。
- 连接超时、读取超时和总超时。
- 哪些错误适合重试，以及指数退避的作用。
- Top-K 对召回率、延迟和后续处理成本的影响。

当前限制：

- 完全依赖 arXiv 的 relevance 排序。
- 没有重试、限流和查询缓存。
- 没有多数据源融合和二次重排序。

## 原理二：Atom XML 与命名空间

arXiv 返回 Atom XML。Feed 声明了默认命名空间，因此解析时不能只搜索 `entry`，需要建立 `atom` 前缀到 Atom URI 的映射，再使用 `atom:entry`、`atom:author/atom:name` 等路径访问元素。

当前解析过程还完成了：

- 合并标题和摘要中的多余空白。
- 提取多个作者。
- 根据链接属性区分摘要页面与 PDF。
- 从发布时间中提取年份。
- 删除 arXiv ID 尾部的版本号。

需要深入掌握：

- XML 元素、属性、层级和命名空间。
- `findtext()` 与 `findall()` 的差异。
- 外部数据字段缺失或格式损坏时的处理策略。
- 为什么解析器应与网络客户端分离。

## 原理三：数据标准化与稳定身份

外部数据不能直接泄漏到后续模块。解析器把 arXiv 数据转换为统一的 Pydantic `Paper` 模型，使检索、持久化、渲染等模块共享同一份数据契约。

例如：

```text
http://arxiv.org/abs/2401.00001v1
→ arxiv:2401.00001
```

删除版本号后，同一论文的不同 arXiv 版本可以共享稳定身份。不过，未来接入 OpenAlex 或 Semantic Scholar 后，还需要处理不同来源 ID 指向同一论文的问题。

需要深入掌握：

- DTO、领域模型与数据契约。
- 技术主键、自然键和稳定标识。
- 数据标准化与实体解析的区别。
- Pydantic 模型在边界校验中的作用。

## 原理四：论文去重

当前实现将标题转为小写，把非 ASCII 字母数字字符替换为空格，再利用 `set` 保存已经见过的标题键。对于 n 篇论文，平均查重复杂度接近 O(n)，同时保持首次出现的论文和原有顺序。

当前方案适合 MVP，但存在明显限制：

- 标题轻微改写可能导致漏判。
- 不同论文同名可能导致误合并。
- 当前正则对中文和其他非 ASCII 标题不友好。
- 没有使用 DOI、作者和年份辅助判断。

更完整的分层策略：

1. DOI 完全匹配。
2. 去版本号后的 arXiv ID 匹配。
3. 规范化标题完全匹配。
4. 标题相似度结合作者和年份判断。

需要进一步学习编辑距离、Jaccard、TF-IDF 余弦相似度和向量相似度，并理解误合并与漏合并之间的取舍。

## 原理五：垂直切片与依赖注入

垂直切片的目的不是一次完成所有高级能力，而是尽早打通一个真实用户场景：从 CLI 输入一直到生成可查看的报告。它可以验证模块接口、数据契约和输出结构是否能够协同工作。

计划中的 `run_pipeline()` 通过 `search_fn` 参数接收检索函数。测试可以注入 `fake_search`，避免依赖真实网络。这是函数级依赖注入，也是控制反转的简单形式。

需要深入掌握：

- Pipeline 编排与具体业务处理的职责边界。
- 依赖注入、控制反转和可测试性之间的关系。
- 单元测试、集成测试与端到端测试的区别。
- 中间产物对审计、复现和断点恢复的价值。

## 实验清单

## 面试表达草稿

> 我在 MOMO Scholar 中实现了论文检索的最小垂直切片。系统将用户研究问题转换为 arXiv API 查询，解析 Atom XML，并把外部字段标准化为统一的 Pydantic `Paper` 模型。检索结果通过规范化标题进行稳定去重，再交给 Pipeline 输出结构化论文数据和初始报告。为了提高可测试性，我分离了网络访问和 XML 解析，并让 Pipeline 支持注入 fake retriever。当前实现属于 MVP，后续会增加基于 DOI 和 arXiv ID 的多级实体解析、重试限流、缓存、混合检索和重排序。
