## 一、MOMO Scholar 解决什么问题

### 原始回答

> MOMO Scholar 主要解决这个论文阅读不可自动化，以及论文引用内容不可溯源的问题。

### 准确表述

MOMO Scholar 主要解决科研调研中论文检索、阅读整理和跨论文综合效率低，以及生成结论缺少可靠引用链、难以回溯原始证据的问题。

> MOMO Scholar 是一个可追溯的论文研究系统。它自动完成论文检索、文本处理、证据提取和综述生成，并让报告中的结论能够回溯到具体论文和原始文本块。

### 核心原理

- **自动化**：减少搜索、整理、筛选和综合论文的重复劳动。
- **可追溯**：让报告中的结论能够沿引用关系返回原始证据和论文来源。

“自动化论文研究”不等于完全替代人工阅读。系统负责缩小阅读范围、组织证据和产生候选结论，重要结论仍需要用户核验。

### 常见误区

- 系统能够完全替代研究人员阅读论文。当前系统主要辅助检索、整理、证据定位和综述生成。
- 报告带有引用编号就代表引用可靠。编号存在只是引用链完整性的一个条件，还要判断引用内容是否支持结论。
- 项目调用了模型，所以一定是 Agent。当前核心流程按固定步骤执行，更准确地说是研究 Workflow。

### 面试追问

1. MOMO Scholar 与普通的“让大模型总结论文”有什么区别？
2. 为什么不能让模型直接阅读所有论文并生成答案？
3. 可追溯性对科研综述有什么价值？
4. 当前系统能够在多大程度上替代人工阅读？

## 二、项目的核心数据流

```text
研究问题
→ 搜索 arXiv 论文
→ 标准化与去重
→ 加载论文文本
→ 切分 Chunk
→ 检索 Evidence
→ 分析单篇论文
→ 综合 Claim
→ 检查引用关系
→ 输出 evidence.json 和 report.md
```

核心数据关系是：

```text
Paper
  ↓ 加载正文并切分
Chunk
  ↓ 针对当前问题检索和排序
Evidence
  ↓ 被报告结论引用
Claim
```

## 三、Paper 与 Chunk

### 原始问题

> Paper 和 Chunk 有什么区别？为什么不能直接拿整篇 Paper 做检索？

### 原始回答

> Paper 是一整篇的论文，Chunk 是切下来的一个一个的小块。整篇 Paper 来检索文章太大，会导致重点不突出。Chunk 可以更好地检索到相关内容，也可以省 Token。

### 准确表述

- `Paper` 是论文级领域对象，主要保存标题、作者、摘要、年份和来源链接等元数据。在当前项目中，它不等同于完整论文原文。
- `Chunk` 是从论文文本中切分出的稳定内容单元，通常保留 `chunk_id`、`paper_id`、章节、页码和文本等信息。

不能直接以整篇论文作为检索单元，原因包括：

1. 一篇论文包含多个主题，整篇计算相关度会稀释局部重点。
2. 返回整篇论文只能说明论文可能相关，不能定位真正支持结论的原文。
3. 完整正文会增加上下文 Token、延迟和模型调用费用。
4. 文本可能超过 Embedding 或生成模型的输入限制并被截断。
5. 论文级结果难以提供精确的章节、页码和文本块引用。

### 常见误区

- `Paper` 对象一定包含完整 PDF 正文。当前 `Paper` 主要承担论文级数据契约；正文由文本加载器获取。
- Chunk 越小越精确。过小会破坏语义完整性，过大又会混合多个主题。
- Chunking 只是为了节省 Token。它还影响召回质量、证据定位和引用可追溯性。

### 面试追问

1. Chunk 太大或太小分别有什么问题？
2. 为什么需要保存稳定的 `chunk_id`？
3. 固定长度切分和按章节切分有什么区别？
4. Chunk overlap 有什么作用，又会带来什么代价？

## 四、Chunk 与 Evidence

### 原始问题与理解过程

> Chunk 是原始素材被切下来的一个一个整块，Evidence 存储了引用所对应的 Chunk 位置，可以用于表明来源。并非每个 Chunk 都是 Evidence。

随后形成的理解是：

> Chunk 经过排序和检索之后转换为 Evidence，两者通过 chunk ID 相互关联。

### 准确表述

- `Chunk` 是相对稳定、与具体研究问题无关的文本单元。
- `Evidence` 是针对当前研究问题，从全部 Chunk 中检索出来的证据候选。
- Evidence 保留 `chunk_id`，并增加与本次检索有关的信息，例如相关性分数和 `run_id`。

不是“单个 Evidence 是 Chunk 的子集”，而是：

> 一次运行产生的 Evidence 集合，通常是全部 Chunk 集合中与当前问题相关的子集。

同一个 Chunk 面对不同问题，可能成为 Evidence，也可能不被选中。

### 常见误区

- 每个 Chunk 都是 Evidence。只有针对当前查询被选中的 Chunk 才会成为 Evidence 候选。
- Evidence 一定来自向量检索。Evidence 也可以来自词法检索、混合检索或其他召回方法。
- 成为 Evidence 就代表一定能支持 Claim。检索阶段只能产生相关候选，不能保证语义支持关系。

### 面试追问

1. 为什么 Chunk 相对稳定，而 Evidence 与查询相关？
2. 为什么 Evidence 还需要保存 `run_id`？
3. 同一个 Chunk 能否在多次运行中成为不同 Evidence？
4. 检索分数高是否意味着它一定能支持最终 Claim？

## 五、Evidence、Claim 与 Citation Checker

### 原始问题

> Citation Checker 是什么？我的猜测是，因为 Evidence ID 只能代表对应的块存在，并不能代表里面的内容和结论正确。

### 准确表述

`Claim` 是报告中的结论性陈述。Claim 应当引用一个或多个 Evidence ID，说明结论的证据来源。

Citation Checker 是引用关系检查器，主要验证：

- Claim 引用的 Evidence ID 是否存在；
- 引用是否来自当前运行；
- Evidence ID 是否满足唯一性；
- Claim 的引用结构是否合法；
- 缺少有效证据时是否应标记为 unsupported。

Citation Checker 检查的是 Claim 与 Evidence 之间的结构关系，而不是直接证明 Claim 的语义和事实正确性。

### 三个不同层次

1. **引用有效性**：引用对象真实存在，ID、唯一性和运行范围正确。
2. **语义支持性**：Evidence 的内容是否真正支持 Claim，也称为 Claim–Evidence entailment。
3. **事实正确性**：Claim 是否准确反映论文和客观事实，是否存在误读、过度概括或信息过时。

例如：

```text
Claim：向量检索一定比词法检索准确。
Evidence：本文使用向量数据库保存论文文本。
```

Evidence 可以真实存在，引用 ID 也可以正确，但这段文字并没有支持“向量检索一定更准确”。

### 常见误区

- Citation Checker 通过就代表 Claim 正确。通过只能证明引用结构满足当前规则。
- Evidence 与问题相关就一定支持 Claim。相关性与支持性是不同判断。
- 论文中的句子一定是客观真理。论文也有适用范围、实验限制和潜在错误。

### 面试追问

1. Citation validity 和 Claim faithfulness 有什么区别？
2. 如何自动评估 Evidence 是否支持 Claim？
3. 为什么不应该只用 LLM Judge 判断 Claim 是否正确？
4. 如何设计 supported、partially supported、unsupported 的标注规则？

## 六、词法检索与向量检索

### 原始问题

> 什么是词法检索，什么又是向量检索？通过关键词检索得出的 Chunk 可能与结论并不相关，对吗？

### 词法检索

词法检索主要根据查询和文本之间的字词重合程度判断相关性。

优点：

- 不依赖 Embedding 模型和向量数据库；
- 成本低、速度快、容易解释；
- 输入固定时容易得到稳定结果；
- 适合作为离线基线和外部服务失败时的降级路径。

限制：

- 不擅长识别同义词和改写；
- 跨语言匹配能力弱；
- 关键词出现不代表内容真正相关；
- 可能遗漏没有共享关键词但语义相近的内容。

### 向量检索

向量检索使用 Embedding 模型把问题和 Chunk 转换成数值向量，再根据余弦相似度等指标寻找语义上接近的内容。

```text
问题文本 → Embedding → 查询向量
Chunk 文本 → Embedding → 文档向量
查询向量与文档向量 → 相似度计算 → 排序
```

优点：

- 能发现同义词、近义表达和语义改写；
- 更适合自然语言问题；
- 没有直接关键词重合时，也可能召回相关内容。

限制：

- 需要 Embedding 模型和向量存储；
- 带来模型、存储、延迟和运维成本；
- 结果不如关键词匹配容易解释；
- 语义相似仍不代表 Evidence 能直接支持 Claim。

### 核心区别

| 对比维度 | 词法检索 | 向量检索 |
|---|---|---|
| 主要依据 | 字词重合 | 向量空间中的语义相似度 |
| 同义表达 | 较弱 | 较强 |
| 精确关键词 | 较强 | 有时不够稳定 |
| 外部模型 | 通常不需要 | 需要 Embedding 模型 |
| 可解释性 | 较强 | 较弱 |
| 基础成本 | 较低 | 较高 |

入门阶段可以简化记忆为：

> 词法检索偏向寻找“说法相似”的内容，向量检索偏向寻找“意思相似”的内容。

### 为什么两种检索都可能出错

```text
检索相关性：这段内容与问题是否有关？
语义支持性：这段内容是否能推出 Claim？
事实正确性：Claim 是否准确反映论文和客观事实？
```

词法检索可能因关键词偶然重合产生假相关；向量检索也可能因主题相似返回无法直接支持结论的内容。

## 七、本轮学习的综合结论

```text
Paper：论文级领域对象
Chunk：从论文文本中切分出的稳定内容单元
Evidence：针对当前问题检索出的相关 Chunk
Claim：报告中的结论性陈述，需要引用 Evidence
Citation Checker：检查 Claim 到 Evidence 的引用结构
```

需要长期保留的区分：

```text
引用存在 ≠ 语义支持 ≠ 事实正确
关键词重合 ≠ 内容相关
语义相似 ≠ 能够支持结论
```
